Part A

The protocontent badge + OAuth

Your vision: a small, dismissible protocontent badge overlaid on each shared artifact, like Vercel's preview toolbar — the entry point for sign-in and owner actions. The collision: artifacts are served with CSP: sandbox (opaque origin, no cookies/storage), so a badge injected into the artifact is identity-blind and network-isolated by construction. It literally cannot know who's signed in.

Recommendation. Build the badge as a trusted protocontent "viewer" shell that frames the unchanged artifact in a sandboxed iframe. It's the only way to put the badge on the surface people actually share — the artifact URL — without weakening the opaque-origin isolation. Sequence it so the risky part comes last.

Architecture

Split content serving by a query flag:

  • <space>.protocontent.app/<name> → a trusted first-party viewer shell (tight CSP, frame-src 'self'; the badge rendered in a closed shadow DOM).
  • <space>.protocontent.app/<name>?raw=1 → today's exact artifact bytes, byte-for-byte, with the unchanged sandbox opaque-origin CSP.
  • The viewer frames ?raw=1 in an <iframe sandbox="allow-scripts allow-forms allow-popups allow-modals allow-downloads"> (no allow-same-origin).

The badge lives only in the trusted parent, never in the artifact — so it's genuinely trustworthy chrome, and the artifact stays an identity-blind, network-isolated passive payload (a cross-origin opaque child can't reach the parent DOM). ?raw=1 doubles as the embed escape hatch; bots / Sec-Fetch-Dest: iframe get raw directly to avoid double-framing.

Sign-in flow (popup, not redirect — keeps the artifact on screen)
1
Badge "Sign in" opens a popup to api.protocontent.com/v1/auth/github (the route exists; 503 until secrets set).
2
GitHub → /v1/auth/github/callback sets a session cookie only on api.protocontent.com (SameSite=None; Secure; HttpOnly), then postMessages the opener and closes.
3
Badge (validating event.origin) re-checks identity via fetch('/v1/me?space=…', {credentials:'include'}) → {login, avatarUrl, ownsThisSpace}.
4
First sign-in links github_user_id → project_id (closes the existing OAuth TODO). Owner actions reuse existing endpoints, now also cookie-auth: keep · unpublish · make-index-public · open index (the ?k= link) · history. Report stays available to everyone.
How the sandbox stays intact

The opaque-origin CSP is preserved byte-for-byte by never putting the badge inside the artifact. Identity lives entirely on api.protocontent.com; *.protocontent.app stays cookie-free by design, so the PSL-free cross-subdomain isolation is untouched. The trust signal to the user is the URL bar (a protocontent-served *.protocontent.app page), not the frame nesting. Token discipline: never postMessage a token into the iframe (sandbox requires targetOrigin:'*' = world-readable).

Key risks
#1 — Third-party cookies. SameSite=None cookies are deprecated/partitioned (Safari ITP, Chrome). A credentialed /v1/me fetch from *.protocontent.app is genuinely cross-site and may be blocked. Mitigation (ship from day one): degrade to the popup-postMessage result as short-lived in-memory identity per tab — the popup is first-party to api during the flow, so it always works.
  • CORS must change carefully — /v1/me + owner actions need exact-origin echo (not *) + Allow-Credentials: true, allowlisted to *.protocontent.app. Getting this wrong breaks auth or opens a CSRF/credential-leak hole.
  • Never set an auth cookie on *.protocontent.app (same registrable site as untrusted artifacts). Dismissal state may only be a non-identifying localStorage boolean.
  • /<name> now returns a shell, not raw bytes — keep ?raw=1 prominent for embed fidelity; serve bots/iframe-dest raw.
  • GitHub OAuth app + secrets must be provisioned (your manual step) before Phase 0 can be tested end-to-end.

Part B

How a coding agent reliably uses protocontent

Goal: an agent that, on its own, (A) publishes the artifacts worth publishing, (B) shares the link on every publish, (C) keeps them updated as it iterates, and (D) ends with both the session-index link and each worked-on file's link.

Recommendation. A defense-in-depth stack of 5 layers, shipped as one installable Claude Code plugin (MCP + Skill + Stop hook), plus a portable AGENTS.md + tool-description baseline for non-Claude agents. The principle: reliability tracks who controls execution — the model (probabilistic) vs. the harness (deterministic). So every critical behavior is covered by both a smart layer (judgment) and a deterministic layer (guarantee). The Stop hook is the anchor that turns "usually shares" into "always shares."

How each behavior is doubly ensured
  • Auto-share (B): bridge prints the link → tool desc says "show it" → skill formats it → Stop hook blocks if the link isn't in the final message.
  • Keep-updated (C): skill+instructions say "re-call with the SAME name to update in place" → Stop hook catches a turn that changed the artifact on disk but didn't republish.
  • Index + per-file links (D): skill prescribes the exact "Published" block from list → instructions require both kinds → Stop hook blocks a summary missing either.
  • What to publish (A): skill carries the when/when-not nuance; "do NOT publish source/secrets" lives in all three text layers to prevent over-publishing.
Note — the "hard prerequisite" the workflow flagged is already done. It worried the bridge emits the bare (404-ing) index URL. In fact PR #3 already wired the tokenized ?k= link: publish + list return it (api.ts) and the bridge surfaces it (index.ts uses the server's spaceUrl). Verified. So the agent already shares a working index link — no Phase-0 blocker here.

# Publishing artifacts with protocontent

## When to publish (and when not to)
PUBLISH: HTML reports/plans/specs, dashboards, prototypes, generated docs,
charts/diagrams, screenshots, a built static site (a folder).
DO NOT: repo source, anything to be committed, secrets, large binaries.
When unsure, publish a *rendered report* view, not raw code.

## The workflow (every time)
1. Publish via protocontent:publish_html / publish_folder. Pick a stable,
   descriptive `name` (e.g. `auth-redesign-plan`).
2. SHARE immediately: put the returned markdown link in your reply.
3. ITERATE = UPDATE IN PLACE: revise → publish again with the SAME `name`
   (same URL, live update). Never mint `plan-v2`/`plan-final`. `keep` to persist.

## End-of-task summary (always BOTH kinds of link)
End with a "Published" section, sourced from protocontent:list:
  **Published**
  - Session index (private, shows everything): <space ?k= link>
  - <artifact-name>: <direct public link>
The index link is PRIVATE (token-gated); per-artifact links are PUBLIC. Label them.
Sketch — AGENTS.md snippet (portable, non-Claude agents)
## Sharing artifacts (protocontent)
When you build something a human would open in a browser — an HTML report, plan,
dashboard, prototype, or built site — publish it with the protocontent MCP tools
(publish_html / publish_folder) and share the returned link. To update it, publish
again with the same `name`. At the end of a task, list both the session-index link
(private) and each artifact's direct link. Do not publish repo source or secrets.
Key risks
  • Probabilistic layers can be ignored — only the Stop hook makes the critical path deterministic; tune the skill description via observe-refine-test so it actually triggers.
  • Over-publishing (source, scratch, every screenshot) — "do NOT publish source/secrets" in all text layers; hook scoped to self-contained artifacts only.
  • Stop-hook nags / loops / block-cap — guard with stop_hook_active, the cheap changed-file pre-filter, per-user opt-out, "if in doubt, allow."
  • Name churn → orphaned URLs / broken update-in-place — skill's one-stable-name rule.
  • Private ?k= link leaks if forwarded — label private vs public. (Your call: surface the index link by default, or only on request?)
  • Non-Claude agents get no deterministic guarantee — tool descriptions + AGENTS.md are the best-effort baseline; full stack is Claude Code.

For you

Decisions & open questions

  • Badge: ship the viewer-shell at all? It changes what /<name> returns (shell, not raw). Phase 1 (badge on the index only) is risk-free and may be enough to start — the viewer is the bigger swing.
  • Index link by default? Should the agent always surface the private ?k= index link, or only when asked? (It's a "shows everything" link — convenient but private.)
  • OAuth purpose. Sign-in mainly unlocks owner actions (keep/unpublish/make-public) on content you already own via your token. Worth the 3p-cookie complexity now, or after there are real users?
  • Stop hook aggressiveness. A real guarantee, but it nags. Default-on with opt-out, or opt-in?
  • Prereqs you hold: the GitHub OAuth app + secrets (Part A), and the npm 0.2.0 publish so the improved instructions reach npx users (Part B Phase 1 ships via the bridge).
