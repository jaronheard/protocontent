Part A

How a coding agent reliably uses protocontent

Goal: an agent that, on its own, (A) publishes the artifacts worth publishing, (B) shares the link on every publish, (C) keeps them updated as it iterates, and (D) ends with both the session-index link and each worked-on file's link.

Recommendation. A defense-in-depth stack of 5 layers, shipped as one installable Claude Code plugin (MCP + Skill + Stop hook), plus a portable AGENTS.md + tool-description baseline for non-Claude agents. The principle: reliability tracks who controls execution — the model (probabilistic) vs. the harness (deterministic). So every critical behavior is covered by both a smart layer (judgment) and a deterministic layer (guarantee). The Stop hook is the anchor that turns "usually shares" into "always shares."

How each behavior is doubly ensured
  • Auto-share (B): bridge prints the link → tool desc says "show it" → skill formats it → Stop hook blocks if the link isn't in the final message.
  • Keep-updated (C): skill+instructions say "re-call with the SAME name to update in place" → Stop hook catches a turn that changed the artifact on disk but didn't republish.
  • Index + per-file links (D): skill prescribes the exact "Published" block from list → instructions require both kinds → Stop hook blocks a summary missing either.
  • What to publish (A): skill carries the when/when-not nuance; "do NOT publish source/secrets" lives in all three text layers to prevent over-publishing.

# Publishing artifacts with protocontent

## When to publish (and when not to)
PUBLISH: HTML reports/plans/specs, dashboards, prototypes, generated docs,
charts/diagrams, screenshots, a built static site (a folder).
DO NOT: repo source, anything to be committed, secrets, large binaries.
When unsure, publish a *rendered report* view, not raw code.

## The workflow (every time)
1. Publish via protocontent:publish_html / publish_folder. Pick a stable,
   descriptive `name` (e.g. `auth-redesign-plan`).
2. SHARE immediately: put the returned markdown link in your reply.
3. ITERATE = UPDATE IN PLACE: revise → publish again with the SAME `name`
   (same URL, live update). Never mint `plan-v2`/`plan-final`. `keep` to persist.

## End-of-task summary (always BOTH kinds of link)
End with a "Published" section, sourced from protocontent:list:
  **Published**
  - Session index (private, shows everything): <space ?k= link>
  - <artifact-name>: <direct public link>
The index link is PRIVATE (token-gated); per-artifact links are PUBLIC. Label them.
Sketch — AGENTS.md snippet (portable, non-Claude agents)
## Sharing artifacts (protocontent)
When you build something a human would open in a browser — an HTML report, plan,
dashboard, prototype, or built site — publish it with the protocontent MCP tools
(publish_html / publish_folder) and share the returned link. To update it, publish
again with the same `name`. At the end of a task, list both the session-index link
(private) and each artifact's direct link. Do not publish repo source or secrets.
Key risks
  • Probabilistic layers can be ignored — only the Stop hook makes the critical path deterministic; tune the skill description via observe-refine-test so it actually triggers.
  • Over-publishing (source, scratch, every screenshot) — "do NOT publish source/secrets" in all text layers; hook scoped to self-contained artifacts only.
  • Stop-hook nags / loops / block-cap — guard with stop_hook_active, the cheap changed-file pre-filter, per-user opt-out, "if in doubt, allow."
  • Name churn → orphaned URLs / broken update-in-place — skill's one-stable-name rule.
  • Private ?k= link leaks if forwarded — label private vs public. (Your call: surface the index link by default, or only on request?)
  • Non-Claude agents get no deterministic guarantee — tool descriptions + AGENTS.md are the best-effort baseline; full stack is Claude Code.

Part B

The protocontent badge + OAuth

Your vision: a small, dismissible protocontent badge overlaid on each shared artifact, like Vercel's preview toolbar — the entry point for sign-in and owner actions. The collision: artifacts are served with CSP: sandbox (opaque origin, no cookies/storage), so a badge injected into the artifact is identity-blind and network-isolated by construction. It literally cannot know who's signed in.

Shipped & live. The badge is built as a trusted first-party "viewer" shell that frames the unchanged artifact in a sandboxed iframe — the only way to put the badge on the surface people actually share (the artifact URL) without weakening the opaque-origin isolation. The viewer shell, the live-version badge, and minimal GitHub OAuth are now in production. What's deferred: full owner actions in the badge and the heavier 3p-cookie hardening.

Architecture (shipped)

Content serving is split in serve.ts by who's asking, not a query flag:

  • A top-level browser navigation to an artifact entry (<space>.protocontent.app/<name>, detected via Sec-Fetch-Dest: document) returns a trusted first-party viewer shell.
  • Everything else — ?raw=1, iframe/embed dests, sub-path assets, bots/unfurlers (no Sec-Fetch-Dest), and HEAD — returns the artifact bytes byte-for-byte with the unchanged opaque-origin sandbox CSP.
  • The framed raw entry sits at the same base path, so relative-asset resolution is preserved.
  • The shell frames the artifact in <iframe sandbox="allow-scripts allow-forms allow-popups allow-modals allow-downloads"> — no allow-same-origin, opaque origin intact.

The badge lives only in the trusted shell, never inside the artifact — so it's genuinely trustworthy chrome, and the artifact stays an identity-blind, network-isolated passive payload (a cross-origin opaque child can't reach the parent DOM). The shell's CSP is tight:

default-src 'none'; frame-src 'self'; script-src 'nonce-…';
style-src 'unsafe-inline'; img-src https://avatars.githubusercontent.com data:;
connect-src 'self' https://api.protocontent.com;
frame-ancestors 'none'; base-uri 'none'
The badge (shipped)

Rendered in a closed shadow DOM in the trusted shell (never inside the artifact). It's a small collapsed dot that expands to a Vercel-toolbar-style bar; dismissible, persisted via a non-identifying localStorage boolean.

Live "new version" notification (shipped)

Reuses the existing Space Durable Object WebSocket fanout. The publish path now broadcasts an enriched {type:"changed", name, version}; the badge connects to /__live and shows a "vN published · Reload" pill when the artifact it's framing gets a new version. /__live is gated by the index token, so live updates require either a ?k= in the URL or owner sign-in (see below).

Minimal OAuth (shipped)
1
GitHub sign-in via popup (/v1/auth/github?popup=1) and sign-out.
2
A stateless, HMAC-signed pc_session cookie is set host-only on api.protocontent.com (HttpOnly; Secure; SameSite=None) — it is never sent to *.protocontent.app, so the cookie-free artifact origin is untouched.
3
GET /v1/me?space=<id> (credentialed, exact-origin CORS echo) returns {login, avatarUrl, ownsThisSpace, indexUrl, k}. The index token (k) + index URL are returned only to the verified owner — never to anyone else.
4
Ownership linking: projects gained github_user_id / github_login / github_avatar columns; POST /v1/claim (requires both the project bearer token and the session cookie) links a GitHub identity to a project. The dashboard auto-claims on sign-in.
Ownership-gated index link (the key privacy decision)

The badge shows a "↩ Session index" link only when you arrived with a ?k= token (you came from your own index) or you're the signed-in owner. Anonymous viewers of a bare artifact link get neither the index link nor the token — this closes the leak where a single shared artifact link would otherwise expose the whole session index.

How the sandbox stays intact

The opaque-origin CSP is preserved byte-for-byte by never putting the badge inside the artifact. Identity lives entirely on api.protocontent.com; *.protocontent.app stays cookie-free by design (the pc_session cookie is host-only on api), so the PSL-free cross-subdomain isolation is untouched. The trust signal to the user is the URL bar (a protocontent-served *.protocontent.app page), not the frame nesting.

Known limitation (documented honestly)
Third-party cookies. The cross-site credentialed /v1/me fetch from *.protocontent.app relies on a SameSite=None cookie, which Safari ITP / Chrome 3p-cookie deprecation may block. In that case the badge degrades to "Sign in" and owner-detection won't stick — but the ?k= URL path (arriving from your index) always works everywhere. A popup-postMessage re-check is wired but subject to the same 3p-cookie limits.
Alternatives considered
OptionVerdict
Trusted viewer shell framing raw bytesShipped — the badge on the shared artifact URL, opaque-origin isolation intact. Split is by Sec-Fetch-Dest, so embeds/bots/assets still get raw bytes byte-for-byte.
Inject badge into artifact bytes (watermark)Dead on arrival — CSP-blocked from auth, mutates user bytes, sits in attacker-controllable DOM (a hostile artifact could spoof/cover it).
Edge-gate auth at Cloudflare, no in-page UIRejected for the badge — strong isolation but delivers no badge/branding. The ?k= private index keeps the edge-gate idea.
Top-level redirect sign-inRejected — blows away the artifact the user was looking at. Popup + postMessage instead.
CHIPS / partitioned cookie as the primary 3p-cookie fixDoesn't solve it — CHIPS partitions by top-level site (the artifact subdomain), so it wouldn't share across spaces. Deferred as an attempt; the ?k= URL path is the reliable everywhere-fallback.

For you

Decisions & open questions

  • Badge: ship the viewer-shell at all? Resolved — shipped. A top-level navigation to /<name> returns the trusted shell; everything else (embeds, bots, assets, ?raw=1) gets the artifact bytes byte-for-byte.
  • Index link by default? Resolved — gated. The ?k= session-index link is shown only to the signed-in owner or someone arriving with a ?k= token; never to an anonymous bare-link viewer.
  • OAuth purpose. Resolved. Minimal sign-in/out + the ownership-gated index link shipped. Full owner actions in the badge (keep/unpublish/make-public/history via the cookie session) are deferred.
  • Stop hook aggressiveness. A real guarantee, but it nags. Default-on with opt-out, or opt-in?
  • Prereqs: the GitHub OAuth app + secrets (Part B) are provisioned and OAuth is live; the bridge 0.2.0 is published, but the improved instructions / skill / hook / plugin that reach npx users (Part A) are not shipped yet.
