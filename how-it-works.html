protocontent · how it works

How protocontent works

A coding agent writes a file; protocontent gives it a live, shareable, sandboxed URL — and one private page per session. Here's the whole machine, end to end.

Control · protocontent.com Content · *.protocontent.app Stack · Cloudflare Workers · R2 · D1 · Durable Objects

The model

It's write_file, but the file lands at a URL

Your agent already writes files. protocontent is the same primitive — write a file — except it lands at a shareable URL instead of in your repo. It's for everything an agent produces that isn't meant to be committed: plans, prototypes, dashboards, screenshots, generated images. HTML first because it's the richest, but content-agnostic by design.

// the agent calls one MCP tool
publish_html({ path: "./plan.html", name: "plan" })

// → returns a tappable link + a live session page
{ url: "https://quiet-harbor-3kf9….protocontent.app/plan",
  spaceUrl: "https://quiet-harbor-3kf9….protocontent.app/?k=…" }

Architecture

Bridge → Worker → storage → live

1
The bridge (runs next to your agent)
npx protocontent is a stdio MCP server that runs wherever your agent runs — laptop or cloud VM. It reads the file/folder off the local disk and uploads the bytes. (A remote server can't read your disk — no MCP primitive allows it — so a co-located bridge is the only way.)
2
The Worker (Cloudflare, one global edge program)
Dispatches by hostname: api.protocontent.com (the control API the bridge calls), *.protocontent.app (serving content), mcp.protocontent.com (a remote-MCP fallback).
3
Storage (R2 + D1)
File bytes go in R2 (object storage); spaces, artifacts, versions, and tokens live in D1 (SQLite). Re-publishing the same name bumps a version; history is kept.
4
Live (Durable Objects)
One Durable Object per space fans out a WebSocket "changed" event to anyone viewing the session page, so it updates the instant the agent publishes. The publish itself doesn't block on it (fire-and-forget) — first publishes stay ~340 ms, not ~1 s.

URLs, spaces & privacy

The capability model

Space = one agent thread

Each thread gets a space at a high-entropy random subdomain (~110 bits, e.g. quiet-harbor-3kf9…). It's cached per-session so the thread is stable, but it's never derived from anything public — the subdomain is the access capability, so it must be unguessable.

Artifacts: open by capability

An individual artifact at <space>/name is reachable by anyone with the link — sharing one artifact is intentional. Knowing the unguessable subdomain is the credential.

The session index is private

The page that lists everything in a thread (/, /__list, /__live) requires the space's index token (?k=). Sharing one artifact link never exposes the whole thread.

Ephemeral by default

Artifacts auto-expire (default 7 days); keep makes one permanent. They're plans and scratch, not products.

Isolation

Cross-artifact isolation without the Public Suffix List

Per-artifact subdomains already give full origin isolation for DOM / storage / scripts. The one thing that leaks across same-site subdomains is cookies. The usual fix is to PSL-list the content domain — but that's a slow, uncertain petition. Instead, every untrusted artifact is served with:

Content-Security-Policy: sandbox allow-scripts allow-forms …  // NO allow-same-origin

The sandbox directive (without allow-same-origin) gives the document an opaque origin — it physically cannot read or set cookies or storage. So an artifact can't write a .protocontent.app cookie that bleeds onto siblings. That's stronger than PSL (opaque-origin per document, not just per site), and it's a response header, not a registry petition. The first-party session page is not sandboxed, so its live WebSocket works.

Safety

Limits, moderation & staying out of git

ControlWhat it does
Size capsReject publishes over 500 files, 25 MB/file, 50 MB total, or a 72 MB body (413).
Rate limitsKV fixed-window: 30 project-mints/hr/IP, 240 publishes/min/IP (429 over).
Kill switchA blocked space serves 410 everywhere; POST /v1/admin/block gated by an ADMIN_TOKEN secret.
ReportsPOST /v1/report (public, rate-limited) records abuse reports.
GitignoreThe bridge keeps .protocontent/ in your repo's .gitignore — published artifacts are ephemeral, not source.

Managing it

The claim dashboard

Visit api.protocontent.com, paste your project token (from ~/.protocontent/config.json, kept in localStorage on the cookie-free control origin), and you get every space your token owns with artifact counts and direct links. GitHub sign-in is wired as a secret-gated opt-in for the future.

One sentence: a co-located bridge writes your agent's files to a Cloudflare Worker, which serves them on unguessable per-session subdomains — sandboxed for isolation, private by token for the index, ephemeral by default, and live as the agent works.
