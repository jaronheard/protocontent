protocontent · framing riff

Your repo is for the software. protocontent is for everything else the agent makes.

A coding agent doesn't just write code. It writes plans, prototypes, design explorations, diagrams, data views — to figure out what the code should be. That material isn't your product, and it doesn't belong in git. Today it has nowhere to live. That is the thing protocontent is actually about.

Why the current framing feels off

It describes the pipe, not the cargo

The framing so far leads with the mechanism: publish the HTML your agent makes to a shareable URL; watch a dead localhost link become one you can open on your phone. All true — but it's a description of the plumbing and one nice benefit. It never names the thing being moved.

The thing being moved is a category we haven't drawn a box around: the stuff an agent produces to do its work, as distinct from the software it ships. Once you see that split, everything else — ephemeral, sandboxed, cross-device, cross-agent — stops being a feature list and becomes an obvious consequence.

protocontent isn't where your agent's code goes.
It's where everything your agent makes to build the code goes.

The core distinction

Two kinds of thing an agent produces

Goes to your repo

The product

The handoff. Reviewed, committed, permanent. What you ship.

  • is source, tests, config, the PR
  • audience teammates, CI, production
  • lifespan permanent, versioned in git
  • judged by does it merge and run
Goes to protocontent

The working material — proto content

The thinking. Looked at, then thrown away. How you get to the product.

  • is plans, prototypes, explorations, diagrams, data views, scratch
  • audience you (often from your phone), and the next agent
  • lifespan ephemeral — true while useful, then gone
  • judged by is it clarifying to look at

The reason a plan.html feels awkward in git isn't a tooling gap — it's a category error. It was never meant to be a handoff. It belongs in the right-hand column, and that column had no home until now.

The analogy that lands it

It's the agent's Figma, not its repo

A designer doesn't commit every frame. They keep dozens of interim boards, variants, and clickable prototypes in Figma — that's how they get to the design. None of it is the deliverable; you never hand a developer your exploration history as the spec. It's scaffolding for thinking, and it's perfectly normal for it to be disposable.

Agents make exactly this kind of throwaway exploration — constantly — they just have no canvas to make it in. So it dies as a file:// path on whatever box happened to run the agent. protocontent is that canvas: the place the agent's prototypes and interim work show up, live, somewhere you can actually look.

Why HTML keeps coming up

"HTML is all you need" — finished

The reason the HTML insight resonates: HTML is the richest medium an agent can write into and a human can just open. A plan, a dashboard, a mockup — the agent renders its thinking, you read it at a glance. But the insight only pays off if you can open the thing, and right now you can't: it's stranded on disk.

And it isn't only HTML. The agent makes whatever artifact is clearest for the job — a diagram, a generated image, a folder of a built static site, a Markdown brief. protocontent is content-agnostic on purpose: it makes the medium the agent thinks in viewable, whatever that medium is. HTML is the headline because it's the best of them, not the whole of them.

The part that's bigger than "see it on your phone"

This material is also context

These files aren't only for you to glance at. A plan one agent writes is something the next agent can read — the same agent tomorrow, a different tool, a teammate's session. Kept somewhere durable and shared rather than locked in one vendor's artifact panel, the agent's working material becomes shared working memory: the explorations, decisions, and prototypes a whole fleet of agents can build on instead of regenerating from scratch.

Not a deliverable. Not source. The working material an agent makes to think — kept, viewable, and readable across every agent you use.

The riff

Headline directions to pull from

★ Lead direction · output vs. working material

Your repo is for the software. protocontent is for everything the agent makes to build it.

Names the category by contrast with the one thing everyone already understands — the repo. Clearest single line.

The Figma angle

Your agent's Figma — for the prototypes, plans, and explorations that aren't the code.

Instantly conveys "exploration you keep, not a handoff you commit." Borrows a mental model people already trust.

The HTML-insight angle

"HTML is all you need" — but only if you can open it.

Anchors to prior art and supplies the missing verb. Good for the audience that already nodded along to Thariq's post.

The not-the-product angle

The agent makes more than code. Where does the rest of it go?

Opens with the gap as a question. The plans, prototypes, and diagrams have had no home — name that.

The shared-memory angle

A home for the files your agents make to think — viewable by you, readable by the next agent.

Leads with the durable, cross-agent context idea rather than the cross-device demo.

The scaffolding angle

Not a deliverable. Not source. The scaffolding your agent builds with — somewhere you can see it.

Defines by negation, which is honest about what it refuses to be (git, a CMS, a permanent host).

The namespace angle

Every platform has user content. Agents make proto content — and it had nowhere to live.

Leans on the name's own etymology (proto + content, the *.usercontent.com pattern) to assert a new category.

Applying it

What to keep, and what to move down

✓Keep the dead-link / phone demo — as proof, not the headline. It shows the working material is now reachable.
↓Demote "publish HTML to a URL." It's the mechanism; lead with the cargo, then explain the pipe.
✓Keep ephemeral / sandboxed / cross-agent — but frame them as consequences of "this is working material, not source."
↓Demote the architecture (Workers, R2, DO). It's a great how-it-works payoff, a weak opener.
✓Add the repo-vs-protocontent split up top. One picture does most of the positioning work.
↓Drop "the missing verb" as the thesis. It's a clever subpoint, not the category.
One line to carry everywhere: protocontent is the home for the working material an agent makes — the plans, prototypes, and explorations on the way to the code — not the code itself.
